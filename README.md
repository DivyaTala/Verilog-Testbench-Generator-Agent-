## Overview

This project automates two core tasks in hardware verification:

1. **Testbench Generation** — Uses Codex (OpenAI CLI) to produce a Verilog/SystemVerilog testbench directly from a human-readable Markdown specification.
2. **Mutant Evaluation** — Compiles and simulates multiple candidate DUT implementations (mutants) using Icarus Verilog (`iverilog` + `vvp`), then selects the one that passes.

If no single winner is found, the system enters an iterative feedback loop: it writes diagnostic notes to `feedback.md`, re-invokes Codex to improve the testbench, and re-runs simulation — repeating until exactly one mutant passes.

---

## Features

- **LLM-generated testbenches** from plain-English Markdown specs
- **Automated compilation and simulation** via `iverilog` + `vvp`
- **Mutant discrimination** — identifies the single correct DUT among candidates
- **Iterative refinement loop** using Codex + structured feedback
- **Cross-platform** — works on Linux, macOS, and Windows (tools must be on `PATH`)

## Workflow

### Step 1 — Add Your Spec

Place a Markdown file describing your design in the `specs/` directory:

```
specs/my_spec.md
```

The spec should describe the DUT's interface, behavior, and any edge cases in plain English. Codex reads this file directly.

---

### Step 2 — Add DUT Candidates (Mutants)

Place one or more candidate implementations in the `mutants/` directory:

```
mutants/mutant_0.v
mutants/mutant_1.v
mutants/mutant_2.v
```

Each file represents a different implementation of the same design. Only one should be functionally correct — the goal of the pipeline is to find it.

> **Note:** Mutants can also be directories if the design spans multiple files.

---

### Step 3 — Generate a Testbench

Run the testbench generator, pointing it at your spec:

```bash
python generate_testbench.py \
  --spec specs/my_spec.md \
  --outdir generated \
  --tb-name tb_my_design \
  --top-module my_dut
```

**Outputs:**

| File | Description |
|---|---|
| `generated/tb_my_design.sv` | The generated SystemVerilog testbench |
| `generated/README_TESTBENCH.md` | Auto-generated notes about the testbench |

---

### Step 4 — Run Mutant Evaluation

Pass the testbench and all candidate mutants to the harness:

```bash
python tools/mutant_harness.py \
  --repo-root . \
  --tb generated/tb_my_design.sv \
  --tb-top tb_my_design \
  --spec specs/my_spec.md \
  --candidate mutants/mutant_0.v \
  --candidate mutants/mutant_1.v \
  --candidate mutants/mutant_2.v
```

**What the harness does:**

1. Compiles each mutant with `iverilog`
2. Runs simulation with `vvp`
3. Scans output for the `TB_PASS` token
4. Stores all results under `runs/`
5. Reports the winning mutant (or triggers refinement if needed)

## Iterative Refinement

If the first round of simulation results in **no winners** or **multiple winners**, the system enters a refinement loop:

```
Generate Testbench
       │
       ▼
  Run Mutants
       │
  ┌────┴────┐
  │ Pass?   │
  └────┬────┘
       │ No single winner
       ▼
Write feedback.md  ──→  Invoke Codex  ──→  Re-run Mutants
       ▲                                         │
       └─────────────────────────────────────────┘
                   (repeat until 1 winner)
```

- `feedback.md` is written automatically with diagnostic context (which mutants passed/failed and why)
- Codex uses this file to improve the testbench in the next iteration
- The loop continues until exactly one mutant is identified as correct

---

## Testbench Requirements

Any testbench used with this framework — whether generated or hand-written — must satisfy these rules:

| Requirement | Detail |
|---|---|
| **Pass token** | Print exactly one line containing `TB_PASS` when the DUT is correct |
| **Failure signaling** | Use `$error` or `$fatal` on any mismatch |
| **Determinism** | No uncontrolled randomness; seed any random sequences explicitly |
| **No waveform dependency** | Results must be determinable from stdout alone (no VCD inspection) |

**Example pass/fail pattern:**

```systemverilog
if (dut_out !== expected) begin
  $error("MISMATCH at time %0t: got %0b, expected %0b", $time, dut_out, expected);
  $fatal(1);
end
// ...
$display("TB_PASS");
$finish;
```

## Notes

- The `runs/` directory is auto-created and can grow large during iterative refinement. It is safe to delete between runs.
- If Codex CLI fails to authenticate, ensure your OpenAI API key is set in your environment (`OPENAI_API_KEY`).
- For designs with complex multi-file structures, pass mutant **directories** instead of individual `.v` files to `--candidate`.
- The testbench generator works best when the Markdown spec clearly defines: port names and directions, expected behavior per input combination, and any timing or reset requirements.
